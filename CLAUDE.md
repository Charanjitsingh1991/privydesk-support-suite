## 1. What PrivyDesk Is

PrivyDesk is a **multi-tenant SaaS helpdesk / customer-support platform** (B2B). Organizations sign up, onboard, brand their workspace, invite agents, and manage support tickets across channels (web, email, chat widget, planned omnichannel). It is currently a **frontend-heavy SPA backed by Supabase** (Postgres + Auth + Storage + Edge Functions). Version `2.0.0`.

**Tenancy model:** one `organizations` row per tenant; every business table carries `organization_id`; isolation is enforced by Postgres Row-Level Security (RLS).

## 5. Core Domain Model (live schema)

Canonical user/tenant tables (migration `20260126212319...`):

- `organizations` — tenant root (`slug`, `plan`, `status`, `custom_domain`, branding, `email_config`)
- `profiles` — 1:1 with `auth.users`; holds `organization_id` + `role`
- `tickets`, `messages` — ticket lifecycle + threaded conversation
- `subscription_plans`, `subscription_usage` — billing/limits

**Roles enum (`user_role`):** `super_admin | admin | agent | client`. There is **no `developer` role** despite some migration files referencing one.

**Ticket status:** `open | in_progress | waiting_customer | resolved | closed`
**Ticket priority:** `low | medium | high | urgent`

RLS helper functions (SECURITY DEFINER): `get_user_organization_id()`, `get_user_role()`, `is_super_admin()`.

## 6. Auth & RBAC — How It Actually Works

- Supabase GoTrue. `Documentation/architecture/AUTHENTICATION.md` declares **passwordless (magic-link + OTP)** as the canonical, "production-ready" system. **But the code still ships classic email/password too** (`AuthContext.signIn/signUp` via `signInWithPassword`, wired to `/login` + `/signup` page variants) alongside the passwordless `/auth/*` flow. This is a doc-vs-code drift: decide and enforce one canonical flow before GA, and remove the other from the router.
- `handle_new_user()` trigger auto-creates a `profiles` row with `role = 'client'` and **no** organization on signup.
- Onboarding (`src/pages/Onboarding.tsx`) creates the org, then **client-side updates the user's own profile to `role = 'admin'` and sets `organization_id`**.
- Route guards: `ProtectedRoute` (auth + org required) and `OnboardingRoute` (auth, no org) in `src/components/ProtectedRoute.tsx`.

> ⚠️ **This is a privilege-escalation hole.** See `SYSTEM_ARCHITECTURE.md` §Security. The `profiles` UPDATE RLS policy is `USING (id = auth.uid())` with no `WITH CHECK` and no column guard, so a user can self-assign `super_admin` and an arbitrary `organization_id`. **Do not build new features on top of this assumption — fix it first** (move role/org assignment into a SECURITY DEFINER RPC, restrict updatable columns).

## 7. API Surface

- Public REST API: edge function `api-v1` (`supabase/functions/api-v1/index.ts`), `verify_jwt = false`, authenticated by **API key** (`Authorization: Bearer` or `x-api-key`) validated via `validate_api_key` RPC. Resources: `tickets`, `tickets/:id/messages`, `users`, `organization`, `analytics`, `webhooks`.
- Uses the **service-role key** and manually scopes every query by `organization_id` — so correctness of tenant isolation here depends on application code, not RLS. Review every new query for the `.eq("organization_id", organization_id)` guard.
- Permission scopes checked via `hasPermission(scope)` against the API key's `permissions`.

> ⚠️ `api-v1` writes webhooks to tables `webhook_configs` / `webhook_logs`, but the schema defines `webhooks` / `webhook_deliveries`. Webhook delivery is effectively broken until reconciled.

## 9. ⚠️ Known Inconsistencies (read before touching migrations)

The `supabase/migrations/` folder is **not internally consistent**. Do not assume a phase file reflects the deployed DB.

1. **Duplicate/competing migrations.** There is a `20260130_consolidated_phases_1_9.sql` **and** individual `20260130_phase1..9_*.sql` files defining overlapping tables. Running all of them as-is will conflict.
2. **`users` vs `profiles`.** 122 references across 16 phase files point at a `users` table and `users.role` that do not exist (the real table is `profiles`). The separate `20260130_rls_policies_all_phases.sql` correctly uses `profiles`. Phase files reference a non-existent `developer` role too.
3. **Webhook table naming** mismatch (see §7).
4. RLS in phase files frequently uses `organization_id IN (SELECT ...)` subqueries instead of the existing `get_user_organization_id()` helper — inconsistent and slower. Note this subquery style is *also* what `MULTI_TENANT_ARCHITECTURE.md` documents, so it's a design-level choice, not just drift.
5. **Doc-vs-code drift beyond migrations.** Several documented systems are design/UI only, not wired: custom-domain routing (`src/middleware/domainResolver.ts` referenced in docs does **not** exist), Let's Encrypt SSL provisioning, the `subdomain-router` Worker, and the usage-limit triggers/cron (`check_plan_limit`, `increment_ticket_usage`, `reset_monthly_usage` appear as doc snippets — confirm they're actually in a migration before trusting them).
6. **Hosting inconsistency in the docs.** `DEPLOYMENT.md` says Cloudflare Pages; `CUSTOM_DOMAIN_ARCHITECTURE.md` says "Hostinger reverse proxy" and uses IP `76.76.21.21` (a Vercel anycast IP). Live DNS confirms the app is actually on **Cloudflare**. Reconcile the custom-domain doc to the real Cloudflare setup.

**Action:** treat the *first* core migration (`20260126212319...`) + `rls_policies_all_phases.sql` as the trustworthy base. Before shipping, consolidate migrations into one ordered, idempotent, conflict-free set verified against a fresh DB. See `DEVELOPMENT_PLAN.md` Phase 0.

## 10. Conventions

- Path alias `@/` → `src/`.
- Keep tenant scoping on **every** query (`organization_id`); never trust client-supplied org IDs in edge functions.
- New tables: enable RLS, add `organization_id`, write SELECT + write policies using the helper functions, add an index on `organization_id`.
- Sanitize all user HTML via `src/lib/security/input-sanitizer.ts` (DOMPurify) before render.
- Validate input with zod at the boundary.
- Secrets live in Supabase function secrets / Vite env (`VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY`); never commit them.
- Run `npm run lint` (zero warnings) and `npm run test` before opening a PR.

---

## 11. Definition of Done for any change

1. Tenant isolation preserved (RLS + explicit org scoping).
2. No new privilege path that lets a non-admin elevate role or cross tenants.
3. Tests added/updated; lint clean.
4. If schema changed: one ordered migration, idempotent, verified against a fresh `supabase db reset`.
5. Types regenerated (`npm run supabase:types`) if DB changed.
